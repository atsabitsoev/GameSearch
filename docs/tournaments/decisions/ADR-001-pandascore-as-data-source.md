# ADR-001 — PandaScore as Data Source

- **Status**: Accepted
- **Date**: 2026-05-25
- **Owner**: Product / iOS team

## Context

Для модуля «Турниры» нужен источник данных по киберспортивным событиям (CS2, Dota 2 и потенциально другие). Источник должен:
- Покрывать tier-S и tier-A турниры по обеим играм.
- Давать данные о матчах, командах, игроках, стримах.
- Иметь публичный API.
- Иметь бесплатный или дешёвый план для старта.
- Быть юридически чистым для использования в App Store-приложении.

В проекте на момент принятия решения уже была заготовка `PandaScoreTournamentsService`, использующая PandaScore. Также был TODO-комментарий `«Удалить при переходе на реальные турниры через Liquipedia API»`, что свидетельствует о ранее рассмотренной альтернативе.

## Decision

Используем **PandaScore REST API** как единственный источник данных для модуля «Турниры» в MVP и ближайших фазах.

Конкретно:
- Бесплатный план PandaScore (Schedules, Results & Context Data).
- Лимит 1000 req/hour на ключ — обходим через многоуровневое кэширование (см. `07-caching-strategy.md`).
- Эндпоинты `/tournaments`, `/matches`, `/teams`, `/players`, `/lives` и связанные.

Переход на платный план — отдельное решение, оформляется новым ADR при необходимости.

## Alternatives Considered

### Liquipedia (MediaWiki API)
- **Плюсы**: бесплатно, нет лимитов, огромная база, множество игр.
- **Минусы**:
  - Это вики-парсинг, а не структурированный API. Каждая страница — wiki-разметка, которую нужно парсить регулярками или MediaWiki-парсером.
  - Нет нормального эндпоинта «список текущих матчей».
  - Live-результаты обновляются с лагом (после ручного редактирования контрибьюторами).
  - Хрупкость: изменение шаблонов на стороне Liquipedia ломает парсинг.
- **Вердикт**: технически возможно, но требует разработки фактически собственного backend-парсера. Не подходит для MVP.

### HLTV unofficial APIs
- **Плюсы**: бесплатно, нативно для CS2, актуальные данные.
- **Минусы**:
  - Только CS2.
  - Все имплементации неофициальные, HLTV может закрыть в любой момент.
  - Юридически серая зона — HLTV не санкционирует такое использование.
  - Высокий риск отзыва приложения из App Store при жалобе.
- **Вердикт**: рисковано для production-приложения.

### Abios
- **Плюсы**: профессиональный API, широкое покрытие.
- **Минусы**: только enterprise-pricing, нет free-tier.
- **Вердикт**: дорого для MVP.

### Sportradar eSports
- **Плюсы**: уровень коммерческих БК, гарантированное качество.
- **Минусы**: только enterprise, цены не публичные, заведомо дорого.
- **Вердикт**: не подходит для бесплатного pet-проекта.

### Собственный парсинг сайтов (HLTV / Liquipedia)
- **Плюсы**: бесплатно, гибко.
- **Минусы**:
  - Юридически серо.
  - Технически хрупко.
  - Требует постоянной поддержки парсера.
  - Высокая нагрузка на сторонние ресурсы — этически плохо.
- **Вердикт**: отвергнуто.

## Consequences

### Положительные
- ✅ Быстрый старт: API уже имеет SDK / документацию, заготовка кода в проекте.
- ✅ Расширяемость: 13 игр поддерживается одним API, добавление новой = новый case в enum.
- ✅ Качество данных: PandaScore — известный игрок, данные надёжны.
- ✅ Юридическая чистота: использование разрешено лицензией для информационных приложений.

### Отрицательные
- ❌ Лимит 1000 req/hour — требует серьёзной работы с кэшем (см. ADR-002).
- ❌ Free-plan **запрещает** беттинг/фэнтези — навсегда отрезает эти направления без перехода на коммерческий контракт (см. ADR-005).
- ❌ In-game live stats (frames, events) — только Pro Live Plan от ~1000€/мес.
- ❌ Платный план дорогой: 400€/мес за игру за Stats Plan, 1000€/мес за Pro Live.
- ❌ Зависимость от внешнего вендора: его падение = падение нашего приложения.

### Митигации
- Серверный кэш через Firestore-прокси (ADR-002) снимает нагрузку и даёт устойчивость к коротким API-сбоям.
- Domain-модели **отдельные** от DTO позволяют в будущем сменить источник без переписывания UI.
- Архитектура `<Entity>ServiceProtocol` позволяет подменить имплементацию.

## Follow-ups

- ADR-002: стратегия серверного кэширования для обхода лимита.
- ADR-005: формализация запрета на беттинг.
- Future ADR: переход на Pro Live Plan PandaScore (если потребуется live in-game data).
- Future ADR: смешанный источник (PandaScore + Liquipedia parsing для расширения coverage) — не раньше Phase 5.
